# P2.0 and P2.1 validation gate outline

This outline makes P2.0 (Pulp command contract) and P2.1 (manifest and custody rules) immediately reviewable and testable.

## Scope and review intent

- In scope: acceptance checks for native Pulp export or import command contract and transfer manifest or custody controls.
- Out of scope: implementation code for later P2.x milestones.
- Boundary rule: low-to-high only; no high-to-low feedback semantics.

## P2.0 command contract acceptance checks

### P2.0-AC-01: CLI capability discovery is explicit
- Verify the selected Pulp image exposes exporter/importer command surfaces and explicitly classify import-check/import status (`available` vs `baseline-blocked`) for the pinned CLI/image pair.
- Capture command help and version outputs as artifacts.
- Fail if command surface is missing, renamed, or plugin-gated with no documented guard and no explicit blocker/remediation gate (for example, required image+CLI upgrade before import rehearsal).

### P2.0-AC-02: Contract fields are pinned and reviewable
- Wash output must identify required inputs and outputs for each command: repository identity, repository version reference, exporter/importer references, task identifiers, and expected artifacts.
- Fail if contract relies on implicit defaults that hide repository version or export identity.

### P2.0-AC-03: Deterministic success and failure semantics
- Define expected exit behavior and error classes for precondition failure, task failure, and timeout.
- Fail if behavior is ambiguous (for example, mixed retry or partial success without explicit status class).

### P2.0-AC-04: Native-first boundary is preserved
- Verify outputs do not introduce a custom Pulp API wrapper or alternate state authority.
- Fail if contract bypasses native Pulp task and repository version evidence.

### P2.0-AC-05: High-side egress assumptions are blocked
- Verify command contract does not require public upstream access on high side.
- Fail if any step references public package sources or public container registry usage on high side.

## P2.1 manifest and custody validation scenarios

### P2.1-AC-01: Manifest minimum schema
- Manifest includes batch ID, direction (`low-to-high` only), created timestamp, source and target context, repository and repository version references, artifact inventory, per-file size, SHA-256, and bounded relative paths.
- Fail if direction or inventory integrity fields are missing.

### P2.1-AC-02: Custody record completeness
- Custody evidence includes operator handoff chain, transfer medium ID, timestamps, disposition, and audit references.
- Fail if custody is not attributable end-to-end.

### P2.1-AC-03: Pre-import integrity enforcement
- Validate path containment, file inventory completeness, size and checksum checks before import.
- Fail closed before import when validation fails.

### P2.1-AC-04: Authenticity gate with waiver handling
- Require detached signature verification for manifest or evidence index before high-side import.
- If signature is missing or invalid, import is blocked unless an explicitly approved waiver exists with expiry and compensating controls.

### P2.1-AC-05: Append-only auditability
- Every override, waiver, rejection, and import disposition creates an append-only audit artifact.
- Fail if waiver usage or rejection outcomes are not recorded.

## Mandatory negative cases (release-blocking)

- P2.NEG-01 Wrong direction (`high-to-low`, bidirectional, or undefined direction).
- P2.NEG-02 Path traversal (`../` or equivalent escape).
- P2.NEG-03 Checksum mismatch (declared versus observed SHA-256).
- P2.NEG-04 Any symlink present in transfer inventory (Phase 2 rejects symlinks by policy).
- P2.NEG-05 Missing export file referenced by manifest.
- P2.NEG-06 Public high-side egress reference (public package source or registry in high-side plan/config).
- P2.NEG-07 Missing signature with no approved waiver.
- P2.NEG-08 Invalid signature with no approved waiver.
- P2.NEG-09 Waiver present but invalid (expired, missing approver, or missing compensating controls).
- P2.NEG-10 Hard-link or special-file presence (device, socket, FIFO, non-regular file).

## Evidence requirements and Playwright CLI expectations

- Every validation case (positive and negative) must produce structured evidence under `evidence/<session-id>/`.
- Required package shape: `README.md`, `manifest.json`, and grouped `apt/`, `fixture/`, `logs/`, `pulp/`, `report/`, `screenshots/`.
- Playwright CLI output is mandatory for each case and must be indexed in `manifest.json` with clear artifact descriptions.
- `README.md` must provide the proof chain from command invocation to acceptance outcome.
- Run `harness/local/scripts/validate-evidence-structure.sh` before claiming evidence completeness.
- If no run was executed, state explicitly that this gate document is planning-only and does not assert runtime evidence.

## What is PR-static vs disconnected or local e2e lane

| Check | PR-static review lane | Disconnected or local e2e lane |
| --- | --- | --- |
| P2.0 command contract completeness and native-first boundary | Required | Optional replay |
| Command syntax or help surface snapshot validation from fixtures/mocks | Required when fixture-backed | Optional runtime confirmation |
| Manifest schema validation and direction or path rules | Required | Required |
| Signature or waiver policy rules (presence, format, expiry fields) | Required | Required |
| High-side public egress reference scanning in plans/configs | Required | Required runtime denied-egress proof |
| Checksum or symlink or missing-file failure semantics | Fixture-based checks required | Required against real transfer bundle |
| End-to-end import-check or import or publish or apt client proof | Not sufficient in PR-only lane | Required |
| Playwright evidence capture and structured package validation | Required for executed tests | Required |

## Review gate output expectation (Wash and Simon)

- Wash output is reviewable when P2.0 acceptance checks map one-to-one to explicit command contracts, with deterministic pass/fail semantics.
- Simon output is reviewable when P2.1 scenarios and all mandatory negatives are encoded as fail-closed controls with signature or waiver auditability.
- River can approve only when both outputs are testable without inference and mapped to evidence expectations.
