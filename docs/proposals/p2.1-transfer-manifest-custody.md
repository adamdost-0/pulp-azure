# P2.1 transfer manifest and custody contract

Date: 2026-05-06T20:27:01.226+00:00
Owner: Simon, Security and Compliance Engineer
Scope: Phase 2 low-side to high-side Pulp export/import rehearsal transfer controls

## Purpose

This proposal defines the security contract for moving a native Pulp export package from the low side to the high side during Phase 2. The contract adds boundary, custody, and audit controls around Pulp outputs. It does not replace Pulp export tables of contents, checksums, task records, or import history.

Phase 2 remains a local rehearsal. Production signing, customer PKI, HSM integration, and formal media handling can be deferred only through an explicit waiver record that is visible in evidence.

## Allowed, assumed, and must prove

### Allowed

- Create a low-side transfer manifest for a native Pulp export package.
- Move artifacts only in the `low-to-high` direction through manual staging or removable-media simulation.
- Validate manifest, file inventory, sizes, SHA-256 hashes, signatures or waivers, and custody records before high-side import.
- Keep high-side acceptance and import evidence on the high side or in a manually approved evidence export package.

### Assumed

- Native Pulp export/import artifacts remain the content authority.
- Apt is the Phase 2 package ecosystem unless the product owner expands scope.
- Operators provide role-based approvals and local custody evidence without embedding real identities, credentials, private keys, or passphrases in repository examples.
- High-side Pulp runtime has no public package or image egress.

### Must prove

- The manifest declares `direction: "low-to-high"` and contains no automated high-side to low-side receipt, callback, telemetry, or sync channel.
- Every transferred path is relative, contained under the transfer root, not a symlink, not hard-linked outside the bundle, and not unexpectedly present.
- Every transferred file has a declared byte size and SHA-256 hash that match on the high side before import.
- The manifest has a valid detached signature, or a Phase 2 signing waiver that records approver role, rationale, expiry, and compensating controls.
- Custody evidence ties the low-side export, media preparation, handoff, high-side validation, and import disposition to the same manifest ID.

## Minimal manifest JSON shape

The manifest is intentionally thin. It identifies the batch, points to native Pulp outputs, and defines boundary controls and file inventory. It must not duplicate Pulp's export TOC semantics.

```json
{
  "schemaVersion": "p2-transfer-manifest/v1",
  "manifestId": "p2-transfer-example-001",
  "direction": "low-to-high",
  "createdAt": "2026-05-06T20:27:01.226+00:00",
  "createdByRole": "low-side-export-operator",
  "source": {
    "side": "low",
    "environmentRef": "low-side-session-ref"
  },
  "target": {
    "side": "high",
    "environmentRef": "high-side-session-ref"
  },
  "repository": {
    "ecosystem": "apt",
    "name": "example-apt-repository",
    "repositoryVersionHref": "/pulp/api/v3/repositories/deb/apt/<repository-id>/versions/<version>/",
    "exportHref": "/pulp/api/v3/exporters/core/pulp/<exporter-id>/exports/<export-id>/",
    "exportTaskHref": "/pulp/api/v3/tasks/<export-task-id>/"
  },
  "boundaryControls": {
    "highToLowReceiptAllowed": false,
    "callbackAllowed": false,
    "telemetryReturnAllowed": false,
    "syncAllowed": false
  },
  "pulpOutputs": [
    {
      "kind": "export-toc",
      "path": "pulp/export/toc.json",
      "sha256": "<64 lowercase hex characters>"
    }
  ],
  "files": [
    {
      "path": "pulp/export/artifact-0001.tar",
      "kind": "pulp-export-artifact",
      "sizeBytes": 123456,
      "sha256": "<64 lowercase hex characters>",
      "symlink": false
    }
  ],
  "signing": {
    "status": "signed",
    "signaturePath": "manifest.json.sig",
    "signatureType": "detached",
    "algorithm": "customer-approved-detached-signature",
    "signerKeyRef": "customer-key-reference-not-secret",
    "verificationProcedureRef": "operator-runbook-section"
  },
  "custodyRecordPath": "custody/custody-record.json",
  "evidenceRefs": [
    "evidence/README.md",
    "evidence/manifest.json"
  ]
}
```

## Field requirements

| Field | Requirement |
| --- | --- |
| `schemaVersion` | Required. Use `p2-transfer-manifest/v1` for this contract. |
| `manifestId` | Required. Stable batch identifier. It must not encode private identity or secret material. |
| `direction` | Required. Must equal `low-to-high`. Any other value is a hard failure. |
| `createdAt` | Required ISO 8601 timestamp. |
| `createdByRole` | Required role or operator identifier approved for evidence. Do not store credentials or private personal data. |
| `source.side` | Required. Must equal `low`. |
| `target.side` | Required. Must equal `high`. |
| `repository` | Required. Include ecosystem, repository name, immutable repository version href, export href, and export task href when available. |
| `boundaryControls` | Required. All feedback, callback, telemetry return, and sync flags must be `false`. |
| `pulpOutputs` | Required when Pulp emits TOC, export metadata, task output, or checksum material. Paths must also appear in `files` if transferred. |
| `files` | Required. Complete inventory of transferred files with relative path, kind, byte size, SHA-256, and `symlink: false`. |
| `signing` | Required. Must be `signed` or `waived-for-phase-2`; missing signing state fails closed. |
| `custodyRecordPath` | Required. Relative path to the custody record carried with evidence. |
| `evidenceRefs` | Required. Relative references to low-side, transfer, and high-side evidence indexes. |

## Direction invariant and no-feedback controls

The transfer is one way. The manifest and workflow plan must encode only `low-to-high` movement.

Hard blocks:

- `direction` is absent or is not `low-to-high`.
- Any field allows, configures, schedules, or implies a high-side to low-side automated receipt.
- Any callback URL, webhook, polling endpoint, message queue, telemetry return path, or sync target points from high side to low side.
- High-side import automation writes a status artifact into low-side storage or attempts to notify low-side services.
- Evidence tooling declares success by sending automated high-side results back across the boundary.

Allowed evidence handling:

- A high-side operator may create a local high-side import disposition record.
- A customer-approved manual evidence export may later move selected, signed, redacted evidence out of the high side. That process is outside this Phase 2 automation contract and must not be implemented as a callback.

## Path, symlink, size, and SHA-256 controls

Validation must fail before import when any file rule fails.

Path rules:

- `path` values are relative to the transfer root.
- Absolute paths are forbidden.
- Empty paths, `.` segments as complete path components, `..` segments, repeated separators used for traversal, and control characters are forbidden.
- Paths must normalize inside the transfer root before file reads.
- The validator must compare the discovered file set to the manifest inventory and reject unexpected files and missing files.

Link rules:

- Symlinks are forbidden even if they resolve inside the transfer root.
- Hard links are forbidden unless the target is the same inode already declared inside the transfer root and the implementation can prove containment. Phase 2 should reject hard links by default.
- Device files, sockets, FIFOs, and other non-regular files are forbidden.

Size and hash rules:

- `sizeBytes` is required for every file and must be a non-negative integer.
- `sha256` is required for every file and must be a lowercase 64-character hexadecimal string.
- High-side validation must calculate size and SHA-256 from staged bytes before running Pulp `import-check` or import.
- Any mismatch quarantines the transfer package and blocks import.

## Detached signature and waiver expectations

### Signed path

The target state is a detached signature over the canonical manifest bytes. The signature file is carried next to the manifest and recorded in `signing.signaturePath`.

Required signed fields:

```json
{
  "signing": {
    "status": "signed",
    "signaturePath": "manifest.json.sig",
    "signatureType": "detached",
    "algorithm": "customer-approved-detached-signature",
    "signerKeyRef": "customer-key-reference-not-secret",
    "verificationProcedureRef": "operator-runbook-section"
  }
}
```

The manifest must never contain private keys, passphrases, bearer tokens, CLI profiles, generated Pulp admin passwords, or reusable signing secrets.

### Phase 2 waiver path

If signing is deferred during Phase 2 rehearsal, the manifest may use `waived-for-phase-2`. The waiver is valid only for local rehearsal evidence and must be treated as a blocker for production readiness.

```json
{
  "signing": {
    "status": "waived-for-phase-2",
    "waiver": {
      "waiverId": "p2-signing-waiver-example",
      "approvedByRole": "security-reviewer-role",
      "approvedAt": "2026-05-06T20:27:01.226+00:00",
      "expiresAt": "2026-05-06T20:27:01.226+00:00",
      "rationale": "Local Phase 2 rehearsal before customer signing authority is selected.",
      "compensatingControls": [
        "sha256-verified-on-both-sides",
        "manual-two-person-custody-review",
        "quarantine-on-any-validation-failure"
      ],
      "productionBlocker": true
    }
  }
}
```

Waiver hard blocks:

- Missing waiver when `status` is `waived-for-phase-2`.
- Waiver lacks approver role, rationale, expiry, compensating controls, or `productionBlocker: true`.
- Waiver attempts to approve production import readiness.
- Waiver references a private key, passphrase, secret value, or personal identity that is not approved for evidence.

## Custody record contract

The custody record is a separate JSON document referenced by the manifest. It records who handled the package, what media was used, how the handoff occurred, and what audit evidence proves it.

```json

  "custodyId": "custody-example-001",
  "manifestId": "p2-transfer-example-001",
  "handlingLabel": "customer-approved-handling-label",
  "releaseAuthorityRef": "release-approval-reference",
  "malwareScanRef": "malware-scan-evidence-reference",
  "exportValidationRef": "low-side-export-validation-reference",
  "media": {
    "mediaId": "media-asset-reference",
    "mediaType": "removable-drive",
    "encrypted": true,
    "tamperSealId": "seal-reference",
    "preparedAt": "2026-05-06T20:27:01.226+00:00",

  },

      "locationRef": "approved-boundary-location",

    "redactionPolicyRef": "customer-redaction-policy-reference"
  }
}
```

### Custody fields

| Area | Required fields |
| --- | --- |
| Record identity | `schemaVersion`, `custodyId`, `manifestId`, handling label, release authority reference. |
| Transfer media | Media ID, media type, encryption status, tamper seal or equivalent control, prepared timestamp, prepared-by role. |
| Operator handoff | Sequence, from role, to role, location reference, start and completion timestamps, condition, operator attestation reference, reviewer attestation reference, anomalies list. |
| High-side acceptance | Accepted-by role, accepted timestamp, validation evidence reference, import disposition. This remains high-side evidence and is not an automated low-side receipt. |
| Audit | Append-only audit record reference, recorded timestamp, retention policy reference, redaction policy reference. |

## High-side import preconditions

High-side automation must complete these checks before Pulp `import-check` or import:

1. Transfer root exists in an approved staging location.
2. Manifest parses as JSON and matches the schema or documented contract.
3. `direction` equals `low-to-high`.
4. Boundary controls forbid high-to-low receipts, callbacks, telemetry return, and sync.
5. No callback, webhook, polling, telemetry, or low-side writeback fields exist anywhere in manifest or workflow metadata.
6. Manifest signature verifies, or a complete Phase 2 waiver is present.
7. Custody record exists and references the same `manifestId`.
8. File inventory exactly matches staged files.
9. All paths are contained relative paths.
10. No symlink, hard-link escape, device file, socket, FIFO, or non-regular file exists.
11. Declared size and SHA-256 match each file.
12. Native Pulp TOC or export metadata is present when Pulp emitted it.
13. Plugin and repository compatibility checks from P2.0 pass.
14. High-side configuration contains no public upstream package source URL, public registry pull, or floating image tag.
15. High-side denied-egress evidence is captured or referenced for the run.

## Fail-closed cases

The import path must stop, preserve evidence, and quarantine the staged package when any of these occur:

- Wrong or missing direction.
- Any high-to-low automated receipt, callback, telemetry, or sync control is present or enabled.
- Missing manifest, invalid JSON, invalid schema, or unexpected field that changes boundary semantics.
- Missing custody record or custody `manifestId` mismatch.
- Missing detached signature without a complete Phase 2 waiver.
- Signature verification failure.
- Waiver attempts to approve production readiness.
- Missing file, unexpected file, size mismatch, SHA-256 mismatch, or duplicate file path.
- Absolute path, path traversal, symlink, hard-link escape, special file, or unreadable file.
- Pulp TOC/export metadata mismatch or missing required native Pulp output.
- Pulp `import-check` fails, times out, or returns an ambiguous result.
- Duplicate import conflict cannot be proven idempotent and safe.
- High-side plan references a public package source, public registry, public telemetry endpoint, or floating image tag.
- Evidence package cannot be indexed by `README.md` and `manifest.json`.

## Required evidence artifacts

Every Phase 2 transfer rehearsal must produce reviewer-readable and machine-indexed evidence. The package must preserve the existing structured evidence model and add transfer-specific artifacts.

Low-side evidence:

- Native Pulp export command log with secrets redacted.
- Pulp export task output and export href.
- Pulp TOC, checksum file, or equivalent native output when available.
- Transfer manifest JSON.
- Detached manifest signature or Phase 2 waiver record.
- Low-side manifest validation output.
- Malware or release scan reference when the environment requires it.

Transfer and custody evidence:

- Custody record JSON.
- Media preparation record.
- Tamper seal or equivalent media integrity record.
- Operator handoff attestation references.
- Reviewer attestation references.
- Anomaly report, even when the report states no anomalies.

High-side evidence:

- High-side manifest schema or contract validation output.
- High-side size and SHA-256 verification output.
- Signature verification output or waiver acceptance output.
- Custody validation output.
- High-side denied-egress static and runtime evidence.
- Pulp `import-check` output.
- Pulp import task output and resulting repository version reference.
- Publication and distribution task output.
- Isolated apt client install proof from high-side Pulp only.
- Quarantine record for any rejected package.

Evidence index requirements:

- `evidence/<session-id>/README.md` remains the human-readable entry point.
- `evidence/<session-id>/manifest.json` indexes every non-root evidence artifact.
- Transfer artifacts should use a `transfer/` group or equivalent manifest group label.
- Low-side and high-side artifacts should be distinguishable by group label or directory.
- Evidence must not contain secrets, private keys, passphrases, generated credentials, or unapproved personal data.

## JSON schema placement

This proposal adds `schemas/transfer-manifest.schema.json` because the repository already stores top-level JSON schemas under `schemas/`. The schema validates the manifest shape and hard boundary flags. The custody record shape remains documented here until P2.2 and P2.3 confirm whether custody records become executable inputs or reviewer-only evidence.

## Open user decisions that block production readiness

1. Which boundary process governs real removable-media movement: customer manual process, cross-domain solution, courier, or another transfer authority?
2. Which custody fields are mandatory for the target environment, and which roles must attest to each handoff?
3. Which detached signature mechanism is required for manifests and evidence: GPG, customer PKI, Sigstore or cosign, Azure Key Vault, managed HSM, or another approved process?
4. Is signing required for the next Phase 2 gate, or can the documented Phase 2 waiver be used for local rehearsal only?
5. Which high-side evidence may leave the boundary manually, and what redaction and approval process applies?
6. What no-egress evidence is acceptable for the customer high-side environment?
7. What retention, immutability, and audit-access rules apply to custody records and evidence packages?
8. Which roles and RBAC model exist for low-side operator, transfer custodian, high-side operator, security reviewer, auditor, and break-glass administrator?
9. What duplicate import, partial import, timeout, and rollback semantics does the customer require?
10. Does Phase 2 remain apt-only, or must this manifest support other ecosystems before Azure planning resumes?
